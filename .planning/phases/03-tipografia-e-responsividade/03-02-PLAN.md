  <files>index.html</files>
  <action>
Fazer duas edições no bloco `&lt;style&gt;` do index.html:

**Edição 1 — Bloco @media (hover: none)** (linha ~520):
Adicionar `.hero-bg` dentro do bloco existente:
```css
@media (hover: none) {
  .pc-label { opacity: 1; }
  .hero-bg { top: 0; height: 100%; }
}
```

Razão: `.hero-bg` tem `top: -20%; height: 140%` para dar margem ao parallax. No mobile, onde o parallax para (via isTouch no JS), essas margens são desnecessárias e causam overflow. Usar `(hover: none)` em vez de `(max-width:700px)` porque o critério de parada do parallax é detecção de touch, não breakpoint de largura.

**Edição 2 — Bloco @media(max-width:700px)** (linha ~506):
Adicionar `#hero` com height 100svh dentro do bloco existente, ANTES das regras atuais:
```css
@media(max-width:700px){
  #hero { height: 100svh; min-height: 580px; }
  .hero-cta-wrap{flex-direction:column}
  /* ... restante das regras existentes ... */
}
```

Razão: `100svh` (Small Viewport Height) é a unidade que corresponde à área visível com a barra de endereços do Safari iOS recolhida. Browsers que não suportam `svh` ignoram a regra e mantêm o `100vh` da regra base — fallback automático via CSS cascade. Reduzir min-height de 680px para 580px no mobile para acomodar telas menores (iPhone SE: 667px).

NÃO alterar `background-position` do hero-bg — já está `center center`, correto para fundo estático.
NÃO alterar o JS do parallax — `isTouch.matches` já para o movimento.
